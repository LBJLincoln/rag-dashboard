4. CHUNKING STRATEGY
================================================================================

PRIMARY: Adaptive LLM-based Semantic Chunking (Semantic Chunker V3.1)

Content-Type Rules:
├─ NARRATIVE: 300-500 words, break at subject changes
├─ TECHNICAL: 200-400 words, break at section boundaries
├─ TABULAR: Keep tables intact, group related rows
└─ LEGAL: 500-800 words, never split clauses

FALLBACK: RecursiveCharacterTextSplitter (if LLM fails)
├─ CHUNK_SIZE: 800 chars
├─ OVERLAP: 200 chars
├─ SEPARATORS: [\n\n, \n, . , ! , ? , ; , , , space]
└─ Result: Deterministic chunks, no LLM dependency

VALIDATION:
├─ Min chunk size: 50 characters
├─ Max chunk size: 3000 characters
├─ Oversized chunks split at sentence boundary
└─ Undersized chunks merged with previous

ENRICHMENT: Each chunk gets 25+ fields
├─ id, content, contextual_content, contextual_prefix
├─ topic, section, chunk_index, total_chunks
├─ parent_id, parent_filename, document_title, document_type
├─ quality_score, version, is_obsolete, chunk_method
├─ tenant_id, trace_id, pii_count, created_at
└─ hypothetical_questions (3 per chunk)

QUALITY SCORES:
├─ EXCEL (xlsx/xls): 0.95
├─ PDF: 0.90 | CSV: 0.90
├─ WORD (docx): 0.85 | WORD (doc): 0.80
├─ POWERPOINT (pptx): 0.70 | POWERPOINT (ppt): 0.65
├─ TEXT/Markdown: 0.60-0.65
├─ HTML: 0.55
└─ UNKNOWN: 0.50

7. BATCH PROCESSING & CONCURRENCY
================================================================================

DOCUMENT PROCESSING:
├─ Input: 1 document (S3 Event webhook)
├─ Output: All chunks for that document
└─ Time: ~30-210 seconds per document

DISTRIBUTED LOCKING:
├─ Mechanism: Redis (lock:ingestion:{hash}, TTL: 3600s)
├─ Prevents: Duplicate processing if S3 event fires multiple times
├─ Multiple documents: Can run in parallel (separate locks)
└─ Cleanup: Lock expires automatically or released on completion

CHUNK BATCHING:
├─ Contextual processing: SplitInBatches (adaptive batch size)
├─ Embeddings: Max 100 chunks per API request
├─ Pinecone upsert: All vectors in single call
└─ Retry: 3 attempts, 2 seconds between retries (HTTP nodes)

ERROR HANDLING:
├─ Lock acquisition failure: Return skip response (DUPLICATE)
├─ Chunking failure: Fallback to RecursiveCharacterTextSplitter
├─ Embedding failure: Retry 3x, then chain-continue
└─ HTTP errors: Retry 3x with 2s delay

THROUGHPUT:
├─ Small document (< 10 pages): 5-20 chunks, ~30-90 seconds
├─ Medium document (10-50 pages): 20-100 chunks, ~60-150 seconds
├─ Large document (50-200 pages): 100-500 chunks, ~120-210 seconds
└─ With 3 workers: ~200-400 documents/hour

Rationale: Rich metadata supports filtering, faceted search, quality scoring
