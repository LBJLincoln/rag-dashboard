5. KEY INSIGHTS & PATTERNS
================================================================================

INSIGHT 1: TWO DISTINCT DATASETS, TWO DIFFERENT FAILURE MODES
  Phase 1 (15fev-s3-quant): 50q dataset, 92% accuracy
    - Company financial data (TechVision, GreenEnergy, HealthPlus)
    - Simple metrics: revenue, net income, cash, assets, EPS
    - 4 failures out of 50 (8%): mostly minor data retrieval issues
    - Schema matches perfectly

  Phase 2 (quantitative-finqa): 42q dataset, <5% accuracy
    - FinQA public company financial documents
    - Complex derived metrics: percentages, growth rates, ratios
    - Requires: pension data, employee share data, rental expenses, bond maturities
    - 38 failures out of 42 (90%): cascading schema mismatches

INSIGHT 2: SQL GENERATION IS THE ROOT BLOCKER
  Of 39 failures:
  - 17 (43.6%): Wrong SQL query (generates valid SQL for wrong fields)
  - 11 (28.2%): SQL query returns no data (field doesn't exist)
  - 5 (12.8%): SQL generation crashes (Llama/Gemma)
  - 6 (15.4%): Timeout (from bad SQL)
  ======================
    Total: 39/39 failures (100%) involve SQL generation issue

  CONCLUSION: The SQL generation node is the critical path failure.

INSIGHT 3: MODEL SELECTION MATTERS
  Model Performance:
  - Working:    (unknown, v6b) - 100% on 1q, 40% on 5q, 20% on 10q
  - Failing:    Llama 70B (v6d) - 0% (crashes with [object Object])
  - Failing:    Gemma 27B (v6e) - 0% (crashes + timeouts)
  - Unknown:    Default (v6c) - 20% (slow, only 2 of 10 correct)

  CONCLUSION: Llama and Gemma models are unsuitable for SQL generation.
              Need to test OpenRouter's Llama 3.3 70B (free tier) or revert.

INSIGHT 4: TIMING PROGRESSION
  As dataset size grows, accuracy DECREASES:
  - 1q: 100%, 0%, 100% (high variance, noise)
  - 5q: 0%, 40%, 40%, 0%, 0% (average 16%)
  - 10q: 20% (only 2/10)

  This is NOT expected behavior. Expected: stable accuracy.
  Possible cause: Larger dataset triggers more edge cases / different LLM paths.

  CONCLUSION: Pipeline is brittle. Small changes in input cause large accuracy swings.

INSIGHT 5: INFRASTRUCTURE STRESS SIGNALS
  As tests progressed:
  - Early (1-5q): Mostly SQL errors and no data
  - Middle (5-10q): Added HTTP 503 and timeout errors
  - Late (Gemma): Cascading failures (503, 408, timeouts)

  This suggests: Each test adds load, triggering cascading failures.
  Hypothesis: n8n worker queue is overloaded, database connections exhausted.

  CONCLUSION: Need to add queue management and connection pooling.
